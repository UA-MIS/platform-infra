# Harbor per-team onboarding (P2.2, D-026)

Onboards a team into Harbor keyed on the SINGLE canonical `<name>` slug (D-026:
AppProject = GitHub Team slug = OIDC group suffix `UA-MIS:<name>` = Vault path =
Harbor project — one slug everywhere). Two steps, both driven by `make`:

| Step | What | Idempotent? | Why a Job vs `make` |
| --- | --- | --- | --- |
| 1. project + RBAC | Harbor project `<name>` (private, auto_scan) + map OIDC group `UA-MIS:<name>` → **Developer** | yes (409 = OK) | a **Job** in `harbor` ns (admin creds never leave the cluster) |
| 2. pull robot → sealed imagePullSecret | project-scoped pull robot → `dockerconfigjson` Secret → kubeseal into `<name>-<env>` | re-run regenerates the token | **`make`** — the robot token is **Harbor-generated, one-time**, so it cannot be declarative |

## Step 1 — project + OIDC group mapping (declarative-ish, idempotent)

```bash
make harbor-onboard NAME=<name>
```

This substitutes `__TEAM__` → `<name>` in `onboard-team-job.yaml` and applies the
Job into the `harbor` namespace. The Job waits for Harbor core, creates the
private auto-scan project `<name>`, and maps the OIDC group `UA-MIS:<name>` to the
**Developer** role (`role_id:2`, `group_type:3` = OIDC). Re-running is safe —
"already exists" (HTTP 409) is treated as success.

> A team member who logs in via Dex (carrying the `UA-MIS:<name>` group claim,
> `oidc_auto_onboard:true`) is auto-onboarded into Harbor and sees ONLY project
> `<name>` — the same tenant isolation the ArgoCD `role:<name>` scoping gives.

## Step 2 — pull robot → sealed imagePullSecret (per env, one-time)

CI (Kaniko, §1.3) pushes with a project robot; the team's workload pulls with a
project-scoped pull robot. The robot token is generated by Harbor on creation, so
this is a scripted-but-imperative `make` step (it ends in a `kubeseal` into the
team's namespace overlay, committed to git — the secret never lands unsealed):

```bash
make harbor-robot NAME=<name> ENV=dev   # repeat per env that pulls (dev/staging/prod)
# -> prints the SealedSecret YAML on stdout; redirect into the team's overlay:
#    > ../<name>-app/.../overlays/dev/harbor-pull-sealed.yaml   (then commit)
```

The team's Deployment references `imagePullSecrets: [{name: harbor-pull}]`. The
robot is scoped to project `<name>` with **pull-only** on `repository` — it can
never reach another team's project (registry-side tenant fence, mirrors the
namespace/AppProject fences).

## Onboarding checklist (one line per primitive, all keyed on `<name>`)

1. GitHub child team slug = `<name>` (D-027)
2. tenant AppProject + namespaces (`tenants/team-<name>/`, tenants-appset)
3. ArgoCD `role:<name>` policy block (`argocd-rbac-cm.yaml`)
4. **Harbor project + OIDC Developer mapping** — `make harbor-onboard NAME=<name>`
5. **Harbor pull robot → sealed imagePullSecret** — `make harbor-robot NAME=<name> ENV=<env>`
6. (Phase-2) Vault path `secret/teams/<name>/*`, Backstage ownership

Steps 4–5 are this directory. SEC-010 (advisory): ideally the whole list is one
automation-owned onboarding action — these `make` targets are the seam for that.
